# Docker Self-Hosting Decisions

## Why a dedicated Node runtime was added

The previous Dockerfile ran the development workflow, wrote a fixed `.env` file during image build, and bundled build-time and runtime concerns into one layer. That was not safe or reproducible for production.

This repository still targets Cloudflare Workers first, but Docker self-hosting needs two extra capabilities:

1. A production HTTP runtime that can serve the React Router build without `wrangler dev`.
2. Persistent document storage outside Cloudflare KV.

## Why the Docker runtime uses a small Node server

The server build generated by `react-router build` is an SSR bundle, but the existing repository only had Cloudflare-oriented entrypoints. A dedicated Node runtime avoids depending on `wrangler dev` in production and keeps the container contract simple: build once, run once, inject configuration through environment variables.

## Why document storage falls back to the filesystem

The application currently reads and writes documents through the `KVNamespace` interface. For Docker self-hosting, introducing a database or changing the persistence abstraction would have been a larger architectural change than necessary for this task.

A filesystem-backed adapter now provides the subset of `KVNamespace` methods used by the app:

- `get`
- `put`
- `delete`

It stores documents under `JSONHERO_STORAGE_DIR`, supports TTL expiration, and lets `docker compose` mount a persistent volume without changing route or service interfaces.
